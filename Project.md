# Project.md — RLBigHW 项目完整分析

## 1. 项目概述

本项目使用强化学习（RL）优化 LLM 推理服务在异构 GPU 集群上的资源分配。核心思路：将集群节点划分为 **3 个资源池**（短请求池 / 长请求池 / 混合池），由 RL Agent 在每一步重新配置池参数，以最大化吞吐量、降低延迟、提高资源利用率并满足 SLA。

当前已实现的算法：**Double DQN + Dueling 架构**。

## 3. RL 环境详细分析 — `LLMClusterEnv`

### 3.1 状态空间（14 维连续向量）

| 维度索引 | 含义 | 计算方式 | 范围 |
|----------|------|----------|------|
| 0 | 短请求池最大队列长度 | `max(queues) / 100.0` | [0, +∞) |
| 1 | 短请求池平均 GPU 显存利用率 | `mean(gpu_mem_util)` | [0, 1] |
| 2 | 短请求池活跃节点比例 | `len(active) / num_nodes_total` | [0, 1] |
| 3 | 长请求池最大队列长度 | 同维度 0 | [0, +∞) |
| 4 | 长请求池平均 GPU 显存利用率 | 同维度 1 | [0, 1] |
| 5 | 长请求池活跃节点比例 | 同维度 2 | [0, 1] |
| 6 | 混合池最大队列长度 | 同维度 0 | [0, +∞) |
| 7 | 混合池平均 GPU 显存利用率 | 同维度 1 | [0, 1] |
| 8 | 混合池活跃节点比例 | 同维度 2 | [0, 1] |
| 9 | 等待队列 P95 等待时间 | `percentile(wait_times, 95) / 10.0` | [0, +∞) |
| 10 | 长请求占比 | `count(len>8000) / total` | [0, 1] |
| 11 | 高优先级请求占比 | `count(priority==1) / total` | [0, 1] |
| 12 | 时间特征（小时正弦） | `sin(2π * hour / 24)` | [-1, 1] |
| 13 | 时间特征（小时余弦） | `cos(2π * hour / 24)` | [-1, 1] |

**设计特点**：
- 每池 3 维（队列 / 显存利用率 / 节点比），3 池共 9 维
- 3 维全局工作负载特征（等待时间 / 长请求比 / 高优先级比）
- 2 维周期时间编码（避免 23:00 → 0:00 的不连续性）

### 3.2 动作空间（36 个离散动作）

动作编码为 `action_id ∈ [0, 35]`，通过三重分解映射到池配置：

```
action_id = a * 12 + b * 4 + c
```

| 因子 | 范围 | 含义 |
|------|------|------|
| `a = action // 12` | 0, 1, 2 | **流量分配模式**（gamma 分布），决定请求路由到哪个池 |
| `b = (action % 12) // 4` | 0, 1, 2 | **并发度偏移**（beta 调整量），-1/0/+2 相对基础值 |
| `c = action % 4` | 0, 1, 2, 3 | **资源限制预设**（v/m 组合），控制显存上限和卸载策略 |

**三种流量分配模式** (`a`)：

| a | 短池 gamma | 长池 gamma | 混合池 gamma | 说明 |
|---|-----------|-----------|-------------|------|
| 0 | 0.7 | 0.2 | 0.1 | 偏向短请求池 |
| 1 | 0.2 | 0.7 | 0.1 | 偏向长请求池 |
| 2 | 0.4 | 0.4 | 0.2 | 均衡分配 |

**四种资源限制预设** (`c`)：

| c | v（显存上限比例） | m（卸载阈值） | 说明 |
|---|-----------------|-------------|------|
| 0 | 0.4 | 0.2 | 保守：低显存占用，易卸载 |
| 1 | 0.6 | 0.4 | 适中 |
| 2 | 0.8 | 0.6 | 激进：高显存占用，少卸载 |
| 3 | 1.0 | 0.9 | 极限：几乎不卸载 |

> 注意：短请求池的 v 上限被额外限制为 `min(v, 0.8)`。

### 3.3 池配置参数体系

每个池（short=0, long=1, mixed=2）有 6 个参数：

| 参数 | 符号 | 含义 | 初始值（短/长/混合） |
|------|------|------|---------------------|
| 节点比例 | `alpha` | 分配到该池的节点占比 | 0.4 / 0.4 / 0.2 |
| 并发实例总数 | `beta` | 池内总推理实例数（均摊到节点） | 2 / 1 / 1 |
| 流量比例 | `gamma` | 请求路由到该池的概率权重 | 0.7 / 0.3 / 0.0 |
| 显存上限 | `v` | 单节点可用显存占总量比例 | 0.6 / 1.0 / 0.8 |
| 卸载阈值 | `m` | 超过此值触发 CPU 卸载 | 0.2 / 0.8 / 0.5 |
| 计算缩放因子 | `c` | 推理速度乘数 | 0.5 / 0.9 / 0.6 |

### 3.4 奖励函数

多目标加权奖励，5 项指标加权求和 + 惩罚项：

```
R = w_tp * R_throughput + w_lat * R_latency + w_util * R_utilization
  + w_cost * R_cost + w_sla * R_sla + penalty
```

| 分量 | 公式 | 默认权重 | 含义 |
|------|------|----------|------|
| `R_throughput` | `completed / 10.0` | 0.3 | 吞吐量奖励，完成越多越好 |
| `R_latency` | `-mean(latency) / 500.0` | 0.3 | 延迟惩罚，延迟越高惩罚越大 |
| `R_utilization` | `-mean(|util - 0.7|)` | 0.2 | 利用率偏离 70% 的惩罚 |
| `R_cost` | `-cost / 10.0` | 0.1 | 运营成本惩罚 |
| `R_sla` | `-sla_violations` | 0.1 | SLA 违规直接计数惩罚 |

**惩罚项**：
- 请求被拒但可卸载：`-5.0`
- 请求被拒且不可卸载（OOM）：`-10.0`，同时统计 `oom += 1`

### 3.5 环境核心机制

#### 节点初始化与池分配
- 从 CSV 读取集群配置，按 GPU 型号归类
- 短请求池偏好 4090（低成本），长请求池偏好 A100（高算力），混合池无偏好
- 节点按 `cost_per_hour` 排序后依次分配

#### 请求到达
- 每个 `step` 从工作负载 CSV 批量读取 `step_batch=10` 条请求
- 请求按 `gamma` 分布的概率路由到对应池
- 最小队列节点优先接收

#### 请求准入与卸载
- 计算请求 VRAM 需求：`weight_gb + kv_per_token_kb * total_tokens / 1024`
- 如果超出 `v * gpu_mem_total_gb`，检查卸载阈值 `m`：
  - `m > 0.5`：允许卸载，延迟乘以 `1.5 + (1.0 - m)`
  - `m <= 0.5`：拒绝请求，返回惩罚

#### 请求处理
- 处理速率 = `1000 / ms_per_token * instances * c`
- GPU 显存利用率每步衰减 0.8 倍（模拟请求完成释放）
- 延迟根据卸载策略添加 1.5 倍乘数
- SLA 阈值：200ms

#### 终止条件
- `terminated`：工作负载缓冲区为空且已运行 >1000 步
- `truncated`：超过 5000 步强制截断

## 4. 模型架构详细分析 — Dueling Double DQN

### 4.1 网络结构

```
输入 (state_dim=14)
    │
    ▼
┌─────────────────────────────────┐
│  共享骨干 (Backbone)             │
│  Linear(14, 256) → LN → ReLU → Dropout(0.1)  │
│  Linear(256, 256) → LN → ReLU → Dropout(0.1) │
└───────────┬─────────────────────┘
            │
    ┌───────┴───────┐
    ▼               ▼
┌─────────┐   ┌─────────────┐
│ Value 流 │   │ Advantage 流 │
│ Linear(256,128)│  │ Linear(256,128)│
│ ReLU           │  │ ReLU           │
│ Linear(128,1)  │  │ Linear(128,36) │
└────┬─────┘   └──────┬──────┘
     │                │
     ▼                ▼
  V(s)           A(s,a)
     │                │
     └─────┬──────────┘
           ▼
  Q(s,a) = V(s) + A(s,a) - mean(A(s,:))
```

- **参数量估算**：~110K（骨干 + Value + Advantage）
- **LayerNorm** + **Dropout(0.1)** 用于稳定训练
- **Dueling 架构**优势：分解状态价值和动作优势，更高效学习哪些状态有价值

### 4.2 训练机制

| 机制 | 实现 |
|------|------|
| **Double DQN** | 用 `policy_net` 选动作，`target_net` 评估 Q 值，避免过估计 |
| **经验回放** | 容量 100K，均匀采样，batch_size=256 |
| **目标网络** | 每 10 步 soft update (τ=0.005)，每 50 episode 硬更新 |
| **损失函数** | Smooth L1 Loss（Huber Loss），对异常值更鲁棒 |
| **梯度裁剪** | `max_norm=10.0` |
| **Epsilon 衰减** | 指数衰减 `ε = 0.01 + 0.99 * exp(-step / 20000)` |

### 4.3 超参数汇总

| 参数 | 值 | 说明 |
|------|-----|------|
| `state_dim` | 14 | 状态向量维度 |
| `action_dim` | 36 | 离散动作数 |
| `lr` | 1e-4 | Adam 学习率 |
| `gamma` | 0.99 | 折扣因子 |
| `epsilon_start` | 1.0 | 初始探索率 |
| `epsilon_end` | 0.01 | 最终探索率 |
| `epsilon_decay` | 20000 | 衰减步数 |
| `buffer_size` | 100000 | 经验回放容量 |
| `batch_size` | 256 | 训练批次大小 |
| `target_update` | 1000 | 目标网络更新间隔 |
| `tau` | 0.005 | 软更新系数 |
| `num_episodes` | 2000 | 总训练回合 |
| `max_steps` | 500 | 每回合最大步数 |
| `step_batch` | 10 | 每步到达请求数 |
| `hidden_dims` | [256, 256] | 隐藏层维度 |

### 5.2 工作负载生成器 (`workload_generator.py`)

**请求到达模型**：非齐次泊松过程
```
λ(t) = λ_base * (1 + A * sin(2π * (hour - φ) / T))
```
- `λ_base = 50`，振幅 `A = 0.5`，峰值偏移 `φ = 9`（上午 9 点高峰），周期 `T = 24h`

**LLM 模型分布**：

| 模型 | 权重大小 | KV/Token | 采样概率 |
|------|---------|----------|---------|
| LLaMA3-7B | 14 GB | 2 KB | 50% |
| Qwen-14B | 28 GB | 4 KB | 35% |
| DeepSeek-70B | 140 GB | 16 KB | 15% |

**Token 长度分布**：对数正态分布
- 输入：μ=7.5, σ=1.2, 范围 [50, 4096]
- 输出：μ=6.0, σ=0.8, 范围 [50, 2048]

**优先级**：P1=20%, P2=50%, P3=30%

**SLA 延迟阈值**：
- LLaMA3-7B：200ms 基础 + 0.1 * (input_tokens / 1000)
- Qwen-14B：500ms 基础 + 同上
- DeepSeek-70B：2000ms 基础 + 同上

**数据规模**：训练 50000 条/7 天 / 验证 10000 条/1 天 / 测试 10000 条/1 天

## 6. 训练流程

```
train_dqn.py 主循环：
│
├── 初始化环境 LLMClusterEnv（加载 train 数据）
├── 初始化 DQNAgent（state_dim=14, action_dim=36）
│
└── for episode in range(2000):
    ├── env.reset()
    └── for t in range(500):
        ├── agent.select_action(state)         # ε-greedy 策略
        ├── env.step(action)                   # 执行动作，返回 (obs, reward, done, info)
        ├── replay_buffer.add(...)             # 存储转移
        ├── agent.learn()                      # Double DQN 更新
        ├── agent.soft_update_target(τ=0.005)  # 每 10 步软更新
        └── break if done
    
    ├── 每 50 episode：target_net 硬同步
    ├── 每 100 episode：保存 checkpoint
    └── 写入 training_log.jsonl
```

**训练可视化** (`plot_training.py`)：
- 奖励曲线 / Loss 曲线 / Epsilon 衰减 / 吞吐量 / OOM & SLA 违规 / 汇总仪表盘

## 8. 关键设计决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 环境建模 | 3 池模型（短/长/混合） | 模拟真实 LLM 服务的多优先级调度 |
| 动作空间 | 36 离散动作（3×3×4 分解） | 平衡细粒度控制与搜索效率 |
| 状态编码 | 14 维紧凑向量 | 每池 3 维 + 全局 5 维，信息密度高 |
| 网络架构 | Dueling DQN | 分离状态价值和动作优势，提升样本效率 |
| 更新策略 | Double DQN + 软更新 | 避免过估计 + 训练稳定性 |
| 请求路由 | 基于 gamma 的概率路由 | 灵活的流量分配，由 Agent 学习最优分布 |
| 时间编码 | 正弦/余弦 | 保持周期连续性，00:00 和 23:59 编码接近 |
